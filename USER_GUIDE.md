## 1. 개요

RTL Converter IDE는 **Algorithm / μArch 문서의 변경 사항을 분석**하여 Verilog/SystemVerilog 또는 SystemC(HLS) 소스 코드를 AI가 자동으로 변환해주는 도구입니다.

### 이 도구가 필요한 이유

설계 변경이 발생할 때마다 Algorithm 문서와 μArch 문서를 읽고, 어떤 RTL 파일의 어느 부분을 수정해야 하는지 파악한 뒤 직접 코드를 고치는 작업은 시간이 많이 걸리고 실수가 발생하기 쉽습니다.

RTL Converter IDE는 이 과정을 자동화합니다:

1. AS-IS / TO-BE 문서 쌍을 등록하면 AI가 변경 내용을 분석합니다.
2. 분석 결과를 바탕으로 관련 RTL 파일을 자동으로 선별하고 코드를 변환합니다.
3. 각 변경 사항에 ID(예: `ALGO-001`, `ARCH-002`)를 부여해 문서와 코드 간 추적이 가능합니다.

### 변환 결과의 성격

변환 결과는 **엔지니어가 검토하고 수정하는 중간 산출물**입니다. AI가 코드를 완성하는 것이 아니라, 수작업의 초안을 만들어주는 역할을 합니다.

도구는 원본 코드의 **구조, 들여쓰기, 변수명, 주석을 최대한 보존**하고, 변경이 필요한 부분만 최소한으로 수정합니다.

### 지원 형식

| 출력 언어 | 입력 파일 확장자 |
|---|---|
| Verilog / SystemVerilog | `.v`, `.sv`, `.verilog` |
| SystemC (HLS) | `.cpp`, `.h`, `.hpp`, `.cc` |

### 동작 환경

- **배포 형태**: 단일 HTML 파일 (`rtl_algo_converter.html`) — 인트라넷 환경에서 외부 CDN 없이 동작
- **LLM 연동**: OpenAI Compatible API 서버 (사내 LLM 서버)
- **dev 서버**: Node.js 기반 로컬 서버. **필수 실행** — dev 서버 없이는 LLM 연동, lint 검증, 로그 저장이 동작하지 않습니다.

## 2. 시작하기

### 2.1 dev 서버 실행

RTL Converter IDE는 **dev 서버와 함께 실행**해야 합니다. dev 서버가 없으면 LLM 연동, lint 검증, 로그 저장 기능이 동작하지 않습니다.

프로젝트 디렉토리에서 아래 명령을 실행합니다:

```bash
node rtl-dev-server.js
```

서버가 정상 시작되면 터미널에 포트 번호가 출력됩니다. 이후 브라우저에서 접속합니다:

```
http://localhost:3000
```

> **서버를 종료하지 마세요.** 변환 작업이 진행되는 동안 터미널 창을 열어 둔 상태로 유지해야 합니다.

### 2.2 처음 실행 시 연결 확인

브라우저에서 도구가 열리면 상단 cfg 바에서 LLM 서버 연결 상태를 확인합니다.

1. **Base URL**, **API Key**, **Model** 항목이 자동으로 채워져 있는지 확인합니다.  
   (dev 서버가 `.env` 파일에서 자동으로 로드합니다)
2. **`연결 테스트`** 버튼을 클릭합니다.
3. 버튼 옆 상태 표시가 **`✓ 연결됨`** 으로 바뀌면 준비 완료입니다.

| 상태 표시 | 의미 |
|---|---|
| `—` | 아직 연결 테스트를 하지 않음 |
| `✓ 연결됨` | LLM 서버 정상 연결, 모델 목록 로드 완료 |
| `✕ 오류` | 서버 주소 또는 API Key 확인 필요 |

연결이 확인되면 Model 드롭다운에서 사용할 모델을 선택할 수 있습니다.

> **자동 로드가 안 된다면?** Base URL / API Key / Model이 비어 있거나 `✕ 오류`가 표시되면 관리자에게 문의하거나, [8절 고급 설정](#8-고급-설정)을 참고하여 직접 입력합니다.

## 3. 화면 구성

화면은 크게 4개 영역으로 구성됩니다.

```
┌─────────────────────────────────────────────────────┐
│  cfg 바  (상단 고정 — LLM 설정 · 변환 강도)            │
├─────────────────────────────────────────────────────┤
│  탭 바   1.입력  2.Algo Diff  3.μArch Diff  4.Summary  5.Results │
├─────────────────────────────────────────────────────┤
│                                                     │
│  메인 콘텐츠 영역  (탭에 따라 변경)                    │
│                                                     │
├─────────────────────────────────────────────────────┤
│  하단 바  ← Back │ 다음 → │ ↓ 로그 │ ⚡ 일괄 실행     │
└─────────────────────────────────────────────────────┘
```

### cfg 바 (상단 고정)

모든 탭에서 항상 보이는 설정 영역입니다.

| 항목 | 설명 |
|---|---|
| **Base URL** | LLM 서버 주소 (dev 서버 실행 시 `.env`에서 자동 로드) |
| **API Key** | 인증 키 (자동 로드) |
| **Model** | 사용할 모델 선택. `연결 테스트` 후 드롭다운에서 선택 가능 |
| **연결 테스트** | LLM 서버 연결 확인 + 모델 목록 로드 |
| **변환 강도** | 슬라이더로 AI 변환 파라미터 조절 (① 정확 / ② 재시도 / ③ 탐색) |
| **⚙ 고급** | temperature, seed 등 세부 파라미터 직접 설정 |
| **컨텍스트 예상** | 현재 입력 기준 예상 토큰 수 실시간 표시 |

### 탭 바

작업 단계를 나타내는 5개의 탭입니다. 순서대로 진행하는 것을 권장합니다.

| 탭 | 하는 일 |
|---|---|
| **1. 입력** | RTL 파일, Algorithm 문서, μArch 문서 등록 및 매핑 |
| **2. Algo Diff** | Algorithm 변경 내용 확인 + AI 분석 |
| **3. μArch Diff** | μArch 변경 내용 확인 + AI 분석 |
| **4. Summary** | 변환 대상 RTL 파일 선정 + μArch 섹션 매핑 |
| **5. Results** | 변환된 코드 확인 + 반복 개선 (Iteration) |

### 하단 바

| 버튼 | 설명 |
|---|---|
| **← Back** | 이전 탭으로 이동 |
| **다음 →** | 다음 탭으로 이동 (마지막 단계에서는 `▶ RTL 변환`으로 변경) |
| **↓ 로그** | LLM 대화 로그를 JSON 파일로 다운로드 |
| **⚡ 일괄 실행** | 파일 등록 후 모든 분석과 변환을 자동으로 순차 실행 |

## 4. 전체 워크플로우

전체 흐름은 다음과 같습니다:

```
[탭 1. 입력]
  RTL 파일 업로드
  + Algorithm AS-IS / TO-BE 문서 등록 및 매핑
  + μArch AS-IS / TO-BE 문서 등록 및 매핑
        ↓
[탭 2. Algo Diff]
  Algo 변경 내용 Diff 확인
  + AI 분석 → ALGO-xxx ID 부여
        ↓
[탭 3. μArch Diff]
  μArch 변경 내용 Diff 확인
  + AI 분석 → ARCH-xxx ID 부여
        ↓
[탭 4. Summary]
  Phase 1: 변환 대상 RTL 파일 자동 선정
  Phase 2: 관련 μArch 섹션 자동 매핑
        ↓
[탭 5. Results]
  RTL 변환 실행
  + Lint 자동 검증
  + Iteration으로 추가 개선
```

> **팁:** 처음 사용한다면 단계별로 진행하면서 각 탭의 결과를 확인하세요.  
> 익숙해지면 **⚡ 일괄 실행** 버튼으로 전체 과정을 한 번에 자동 실행할 수 있습니다. ([5.1절 참고](#51-일괄-실행-))

---

### 탭 1 — 입력

모든 파일과 문서를 등록하는 탭입니다. 3개 섹션으로 구성됩니다.

#### ① RTL / SystemC 파일 업로드

1. 파일 영역에 `.v`, `.sv` (Verilog/SV) 또는 `.cpp`, `.h`, `.hpp` (SystemC) 파일을 드래그하거나 클릭하여 업로드합니다.
2. **출력 언어**를 선택합니다:
   - `Verilog / SystemVerilog` — RTL 레벨 변환
   - `SystemC (HLS)` — HLS 레벨 변환
3. 파일이 업로드되면 모듈 간 의존성 그래프가 자동으로 분석됩니다. ([5.3절 참고](#53-모듈-의존성-그래프))

> 여러 파일을 한 번에 올릴 수 있습니다. 변환이 필요한 파일과 참조용 파일을 모두 등록하세요. 변환 대상 파일은 Summary 단계에서 AI가 자동으로 선별합니다.

#### ② Algorithm 문서 등록 및 매핑

Algorithm 문서의 AS-IS(변경 전)와 TO-BE(변경 후) 버전을 등록하고 서로 연결합니다.

1. **AS-IS** 영역에 변경 전 Algorithm 문서를 업로드하거나 텍스트를 직접 붙여넣습니다. (텍스트 파일 업로드 또는 직접 붙여넣기 모두 지원)
2. **TO-BE** 영역에 변경 후 Algorithm 문서를 등록합니다.
3. 각 AS-IS 문서와 대응하는 TO-BE 문서를 **매핑**합니다:
   - **자동 매핑**: `자동 매핑` 버튼 클릭 → AI가 내용을 분석해 자동으로 연결
   - **수동 매핑**: AS-IS 카드에서 TO-BE 카드로 마우스로 드래그하여 직접 연결

#### ③ μArch 문서 등록 및 매핑

Algorithm 문서 등록과 동일한 방식으로 μArch 문서의 AS-IS / TO-BE를 등록하고 매핑합니다.

> **μArch 문서 포맷 안내:** μArch 문서에 테이블이 포함된 경우, 도구의 doc parser가 생성한 인코딩 형식(`[TABLE | ROW]` 형식)을 사용하면 AI가 테이블 내용을 정확하게 해석합니다.

#### ④ 컨텍스트 옵션

탭 하단의 체크박스로 변환 동작을 조정합니다. 기본값으로 모두 켜져 있습니다.

| 옵션 | 기본값 | 설명 |
|---|---|---|
| **TO-BE μArch 전문 포함** | ON | 변환 프롬프트에 TO-BE μArch 원문을 참조용으로 포함 |
| **사전 검토** | ON | 변환 전 AI가 변경 계획을 먼저 검토하는 단계 수행 |
| **Lint 검증** | ON | 변환 완료 후 자동으로 lint 실행 및 오류 시 재시도 |

---

### 탭 2 — Algo Diff

등록한 Algorithm 문서 쌍의 변경 내용을 확인하고 AI 분석을 실행하는 탭입니다.

#### 화면 구성

- **좌측**: AS-IS → TO-BE 라인별 diff 표시
  - 녹색(`+`): 추가된 내용
  - 빨간색(`-`): 삭제된 내용
  - 회색: 변경 없는 내용
- **우측**: AI 분석 결과 (변경점 요약 및 RTL 영향 항목 목록)

등록한 문서 쌍이 여러 개인 경우 상단 탭으로 전환하며 각각 확인할 수 있습니다.

#### AI 분석 실행

1. **`▶ 분석 실행`** 버튼을 클릭합니다.
2. AI가 diff를 분석하고 RTL에 영향을 주는 변경점을 추출합니다.
3. 각 항목에 `ALGO-001`, `ALGO-002` 형태의 ID가 자동으로 부여됩니다.

분석 결과가 마음에 들지 않으면:
- 우측 패널 하단의 **추가 지시사항** 입력창에 보완할 내용을 입력한 후 `재분석` 버튼(또는 `Ctrl+Enter`)으로 재실행합니다.
- **실행 취소 / 다시 실행** 버튼으로 이전 분석 결과로 되돌릴 수 있습니다.
- 분석 중 **`■ 중단`** 버튼으로 언제든 중지할 수 있습니다.

---

### 탭 3 — μArch Diff

Algo Diff 탭과 동일한 방식으로 μArch 문서의 변경 내용을 확인하고 AI 분석을 실행합니다.

- 분석 결과 항목에는 `ARCH-001`, `ARCH-002` 형태의 ID가 부여됩니다.
- 추가 지시사항, 실행 취소/다시 실행, 중단 버튼 모두 동일하게 동작합니다.

---

### 탭 4 — Summary

Algo / μArch 분석 결과를 종합하여 변환 대상을 결정하는 탭입니다. 2개의 Phase로 구성됩니다.

#### Phase 1 — 변환 대상 RTL 파일 선정

AI가 분석 결과와 모듈 의존성 그래프를 참고하여 등록된 RTL 파일을 3가지로 분류합니다:

| 분류 | 의미 |
|---|---|
| **수정 대상** | 실제로 변환을 적용할 파일 |
| **참조용** | 변환 시 컨텍스트로만 참조하는 파일 (직접 수정 없음) |
| **제외** | 이번 변경과 무관한 파일 |

#### Phase 2 — μArch 섹션 매핑

각 변환 대상 파일에 어떤 μArch 섹션이 관련되는지 자동으로 매핑합니다. 이 매핑 결과가 변환 프롬프트의 컨텍스트로 사용됩니다.

#### 분석 실행

**`▶ 분석 실행`** 버튼을 클릭하면 Phase 1, 2가 순서대로 자동 실행됩니다.  
완료 후 **`μArch 섹션 확인`** 버튼으로 매핑 결과를 검토할 수 있습니다.

> **사전 조건:** Summary 분석을 실행하기 전에 Algo Diff(탭 2)와 μArch Diff(탭 3)의 AI 분석이 모두 완료되어 있어야 합니다.

> **분류 결과 수정:** Phase 1의 파일 분류 결과가 의도와 다를 경우 수동으로 변경할 수 있습니다. 각 파일 카드 우측의 **수정대상 / 참조용 / 제외** 버튼을 클릭하여 분류를 변경한 후 변환을 진행하세요.

---

### 탭 5 — Results

RTL 변환을 실행하고 결과를 확인하는 탭입니다.

#### 변환 실행

하단 바의 **`▶ RTL 변환`** 버튼을 클릭하면 Summary에서 선정된 파일들에 대해 변환이 시작됩니다.

> 토큰 수가 128K를 초과하면 경고 모달이 표시되며, 단일 컨텍스트 또는 순차 처리를 선택할 수 있습니다. ([7절 트러블슈팅 참고](#7-트러블슈팅))

#### 결과 확인

변환이 완료되면 파일별 탭으로 결과를 확인합니다.

- 각 탭에 **`✓ lint`** 또는 **`⚠ lint N개 오류`** 배지가 표시됩니다.
- 화면은 좌우로 분할됩니다:
  - **좌측**: 변환된 코드. 변경된 라인에 ID 주석이 표시됩니다:
    - `// ALGO-001:` — Algorithm 문서 변경에서 비롯된 수정
    - `// ARCH-002:` — μArch 문서 변경에서 비롯된 수정
    - `// ITER:` — Iteration으로 추가된 수정
  - **우측**: 변경사항 요약. 각 항목의 출처(ALGO/ARCH/BOTH/ITER), 타입(ADD/MOD/REMOVE), ID 배지, 설명이 표시됩니다.
- 우측 패널의 **ID 배지를 클릭**하면 해당 Diff 탭으로 이동하여 원본 변경 내용을 확인할 수 있습니다.

변환 결과를 바탕으로 추가 개선이 필요하면 **Iteration** 기능을 사용합니다. ([6절 참고](#6-iteration-워크플로우))

## 5. 주요 기능 상세

### 5.1 일괄 실행 ⚡

파일과 문서를 모두 등록한 후 **⚡ 일괄 실행** 버튼을 클릭하면 아래 4단계를 자동으로 순차 실행합니다:

1. Algo Diff AI 분석
2. μArch Diff AI 분석 (1번과 병렬 실행)
3. Summary 분석 (Phase 1 + Phase 2)
4. RTL 변환

실행 전 확인 모달이 표시되며, 확인 후 진행 상황을 오버레이로 실시간 확인할 수 있습니다. 실행 중 **■ 중단** 버튼으로 언제든 중지할 수 있으며, 완료 후 **⚡ 진행 보기** 버튼으로 오버레이를 다시 열 수 있습니다.

> **주의:** 일괄 실행은 사용자 확인 없이 자동으로 변환까지 진행합니다. AI 분석 결과가 부정확하면 변환 결과도 부정확할 수 있으므로, 처음 사용할 때는 단계별로 진행하며 각 분석 결과를 직접 확인하는 것을 권장합니다.

**토큰 자동 관리:** 변환 시 전체 컨텍스트가 128K 토큰을 초과하면 자동으로 순차 처리로 전환됩니다. 단, 256K 이상의 대형 컨텍스트를 지원하는 모델(예: Kimi-K2.5)을 사용 중이라면 초과해도 단일 컨텍스트로 실행됩니다.

---

### 5.2 변환 강도 슬라이더

cfg 바의 슬라이더로 LLM 변환 파라미터를 조절합니다. 상황에 따라 단계를 높여가며 시도하세요.

| 단계 | 표시 | temperature | seed | 언제 사용하나 |
|---|---|---|---|---|
| 1 | **① 정확** | 낮음 | 고정 (42) | 기본값. 항상 동일한 결과를 원할 때 |
| 2 | **② 재시도** | 보통 | 랜덤 | ① 결과가 아쉬울 때, 조금 다른 해석을 원할 때 |
| 3 | **③ 탐색** | 높음 | 랜덤 | 완전히 다른 접근 방식을 시도하고 싶을 때 |

**⚙ 고급** 버튼을 클릭하면 temperature, seed 값을 직접 입력하거나, 추론 모델용 thinking 버퍼를 설정할 수 있습니다.

---

### 5.3 모듈 의존성 그래프

RTL 파일을 업로드하면 자동으로 모듈 간 인스턴스 연결 관계를 분석합니다.

- **Verilog/SV**: `ModuleName #(params) instName (ports);` 패턴 파싱
- **SystemC**: `SC_MODULE` 멤버 선언 및 `.bind()` 패턴 파싱

분석된 의존성 정보는 두 곳에 활용됩니다:

1. **Summary Phase 1**: 변환 대상 파일 선정 시 upstream/downstream 관계를 힌트로 제공하여 AI가 포트 변경의 파급 범위를 파악하도록 합니다.
2. **변환 프롬프트**: 대상 파일의 2-hop 범위 내 연결 파일 정보를 자동으로 포함하여, 포트 변경이 연관 모듈에 미치는 영향까지 고려한 변환이 이루어집니다.

---

### 5.4 Lint 자동 검증

변환이 완료되면 dev 서버를 통해 자동으로 lint를 실행합니다.

**동작 흐름:**
1. 변환 완료 → lint 자동 실행
2. 오류 발견 시 → LLM이 자동으로 수정 재시도 (최대 2회)
3. 결과에 따라 Results 탭의 파일 탭에 배지 표시:

| 배지 | 의미 |
|---|---|
| `✓ lint` | lint 통과 |
| `⚠ lint N개 오류` | 오류가 남아 있음 (2회 재시도 후에도 미해결) |

오류가 남은 경우 우측 패널에서 오류 목록과 라인 번호를 확인할 수 있습니다. Iteration 기능으로 추가 수정을 진행하거나, 변환된 코드를 복사하여 외부 편집기에서 직접 수정하세요.

---

### 5.5 컨텍스트 옵션

탭 1 (입력) 하단에 위치한 3개의 체크박스입니다. 기본적으로 모두 활성화되어 있으며, 특별한 이유가 없으면 그대로 두는 것을 권장합니다.

| 옵션 | 기본값 | 설명 |
|---|---|---|
| **TO-BE μArch 전문 포함** | ON | 변환 프롬프트에 TO-BE μArch 문서 원문을 참조용으로 포함합니다. 변경 맥락을 AI에게 더 풍부하게 전달합니다. |
| **사전 검토** | ON | 변환 전 AI가 변경 계획을 먼저 수립하고 검토하는 단계를 추가합니다. 변환 품질이 향상되지만 시간이 다소 증가합니다. |
| **Lint 검증** | ON | 변환 완료 후 lint를 자동으로 실행합니다. OFF 시 lint 단계를 건너뜁니다. |

## 6. Iteration 워크플로우

변환 결과를 검토한 후 추가적인 수정이 필요하면 Iteration 기능을 사용합니다. Results 탭 하단의 **개선 지시** 패널에서 진행합니다.

### 언제 사용하나

- 변환된 코드의 특정 부분이 의도와 다를 때
- Lint 오류가 자동 재시도 후에도 남아 있을 때
- 변환 강도를 높여 다른 접근 방식으로 재시도하고 싶을 때
- 초기 변환 범위 밖의 추가 수정이 필요할 때

### 사용 방법

1. **개선 지시** 입력창에 원하는 수정 내용을 자유롭게 입력합니다.
   ```
   예) alu.v의 multiply 결과 폭을 64비트로 수정해줘
   예) pipeline_ctrl.sv의 stall 로직에서 ARCH-003 반영이 누락된 것 같아, 다시 확인해줘
   ```

2. **🔍 파일 분석** 버튼을 클릭합니다.
   - AI가 지시 내용을 분석하여 관련 파일을 자동으로 선택합니다.
   - 각 파일의 선택 이유를 확인할 수 있습니다.

3. 파일 목록을 검토하고 필요에 따라 수동으로 추가/제외합니다.
   - **전체** / **해제** 버튼으로 일괄 선택/해제 가능합니다.
   - 예상 토큰 수가 표시되므로 컨텍스트 크기를 미리 확인할 수 있습니다.

4. **▶ 개선 실행** 버튼을 클릭합니다.
   - 선택된 파일들에 대해 수정이 적용됩니다.
   - 변경 사항은 `ITER` 출처로 표시됩니다.

### 버전 관리

Iteration을 실행할 때마다 새로운 버전 탭이 생성됩니다. 버전 탭을 클릭하여 이전 결과와 비교하거나, 원하는 버전을 선택해 이어서 작업할 수 있습니다.

## 7. 트러블슈팅

### 토큰 초과 경고가 표시될 때

변환 컨텍스트가 128K 토큰을 초과하면 경고 모달이 표시됩니다.

**선택지:**
- **단일 컨텍스트로 실행**: 전체 파일을 한 번에 처리합니다. 모델의 컨텍스트 한도를 실제로 초과하면 오류가 발생할 수 있습니다.
- **순차 처리로 실행**: 파일을 하나씩 순서대로 변환합니다. 시간이 더 걸리지만 안정적입니다.

**근본적인 해결 방법:**
- Summary 탭에서 변환 대상 파일 수를 줄입니다. (제외 파일 재분류)
- 더 큰 컨텍스트를 지원하는 모델을 선택합니다. (예: Kimi-K2.5는 256K 지원, 자동으로 단일 처리)

---

### AI 분석 결과를 조정하고 싶을 때

Algo Diff 또는 μArch Diff 탭에서 분석 결과가 기대와 다를 때:

1. **추가 지시사항** 입력창에 보완할 내용을 입력합니다.
   ```
   예) ALGO-003과 ALGO-004는 동일한 변경으로 합쳐줘
   예) 파이프라인 스테이지 변경은 RTL 영향 없으므로 제외해줘
   ```
2. `재분석` 버튼 (또는 `Ctrl+Enter`)으로 재실행합니다.
3. **실행 취소** 버튼으로 이전 분석 결과로 되돌릴 수 있습니다.

변환 강도를 **② 재시도** 또는 **③ 탐색**으로 높이면 다른 관점의 분석 결과를 얻을 수도 있습니다.

---

### Lint 오류가 계속 남을 때

자동 재시도(최대 2회) 후에도 lint 오류가 남은 경우:

1. 우측 패널에서 오류 목록과 라인 번호를 확인합니다.
2. **Iteration 기능**을 사용하여 오류 내용을 개선 지시로 입력합니다.
   ```
   예) alu.v 32번째 줄 lint 오류 수정해줘: undeclared signal 'carry_out'
   ```
3. 오류가 복잡하거나 구조적인 문제라면 변환된 코드를 복사하여 외부 편집기에서 직접 수정하는 것이 빠를 수 있습니다. (도구 내 코드 직접 편집은 지원되지 않습니다)

---

### LLM 서버 연결 오류 (`✕ 오류`)

1. dev 서버가 실행 중인지 확인합니다. (`node rtl-dev-server.js`)
2. cfg 바의 Base URL과 API Key가 올바르게 입력되어 있는지 확인합니다.
3. **`연결 테스트`** 버튼을 다시 클릭합니다.
4. 문제가 지속되면 관리자에게 LLM 서버 상태를 문의합니다.

## 8. 고급 설정

이 섹션은 도구의 동작을 세밀하게 조정하고 싶은 사용자를 위한 내용입니다. 일반 사용자는 기본값으로 사용하면 됩니다.

### rtl-converter-config.js

dev 서버 실행 시 자동으로 로드되는 설정 파일입니다. 주요 설정 항목은 다음과 같습니다.

```js
RTL_CONVERTER_CONFIG = {
  // μArch 문서 청크 분할 전략
  uarchChunk: {
    strategy: 'both',          // 'heading' | 'token' | 'both'
    activeHeadingLevel: 2,     // ## 까지 헤딩으로 분할
    maxChunkTokens: 1500,      // 청크당 최대 토큰
    overlapTokens: 200,        // 청크 간 overlap
  },

  // Summary Phase 2 μArch 섹션 매핑
  uarchMapping: {
    maxSectionsPerFile: 5,     // 파일당 최대 관련 섹션 수
  },

  // 128K 초과에도 단일 컨텍스트로 실행할 모델 목록
  largeCxtModels: [
    'Kimi-K2.5',               // 256K context 모델
  ],
}
```

### LLM 서버 및 모델 변경

cfg 바에서 직접 수정할 수 있습니다. 변경 후 **`연결 테스트`** 버튼으로 연결을 확인하세요.

| 항목 | 설명 |
|---|---|
| **Base URL** | OpenAI Compatible API 서버 주소 (`/v1` 포함) |
| **API Key** | 서버 인증 키 |
| **Model** | 사용할 모델명. 연결 테스트 후 드롭다운 목록에서 선택 가능 |

`.env` 파일을 수정하면 다음 dev 서버 재시작 시 기본값으로 적용됩니다:

```env
BASE_URL=http://your-llm-server/v1
API_KEY=sk-...
DEFAULT_MODEL=your-model-name
```

### ⚙ 고급 파라미터

cfg 바의 **⚙ 고급** 버튼을 클릭하면 변환에 사용하는 LLM 파라미터를 직접 설정할 수 있습니다.

| 파라미터 | 설명 |
|---|---|
| **temperature** | 낮을수록 결정적, 높을수록 다양한 결과 |
| **seed** | 동일 seed는 동일 결과를 재현 (모델이 지원할 경우) |
| **thinking 버퍼** | 추론 모델 사용 시 내부 추론에 할당할 토큰 수 (0 = 비활성) |
